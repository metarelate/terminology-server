Types of Installation

 Several installations of the WMO TDCF Web accessible registers (TDCF-Register) are possible, depending on functionalities provided.
To make a web-representation of codes available, there is no need to install any particular software other than a web server and only a minimum of configuration is required. The terminology server in fact renders all codes simply as files with a proper directory structure. A copy of these files that is performed maintaining the directory structure (e.g.: recursively or via a tar archieve), can be just deployed in a web server directory for "web publication". Clearly, the root of the web server must be compatible with the URLs used for the codes (note that URIs and URLs may differ for testing or other purposes. cfr. Terminology Manager platform design).
To allow for modification of codes (currently via command line), the full Terminology Server must be installed. The Terminology Server requires that an RDF representation from which to build a terminology model is already available.
Such representation is provided essentially by the D2RQ system, following conventions outlined in the Terminology Manager Platform Design document. The D2RQ system complements some functionalities in the Terminology Server to provide the Terminology Mapper.

Requirements

Web-representation of codes

  • A web server supporting content negotation via .var files (e.g.: Apache). The system is tested under: Apache 2.2.22 (Unix)

Terminology Manager

All of the above, plus:

  • A Java runtime system (>1.6). The system is tested under OpenJDK 1.10.6 (amazon built) and Java(TM) SE Runtime Environment (1.6.0, Mac OSX)

Terminology Mapper

All of the above, plus:

  • A D2RQ installation (ref. Terminology Manager platform design). The system is tested under version 0.7
  • A database supported by D2RQ. The system is tested under: mysql 5.1.44
  • (Optional) the utility rapper for synatx conversion. The syste, is tested under version 2.0.8.
In addition to the above, access to a server with permissions to operate on the file system and on web accessible directories is required.

How to install web accessible representations of WMO codes

Make sure the mod_negotation module is installed in Apache.
Edit the httpd configuration file (usually found at /etc/httpd/conf/httpd.con on unix systems) to specify the directory index option as folllows:

DirectoryIndex index.html index.html.var code.var register.var

(it should include code.var and register.var).

Is is then sufficient to copy and expand a tarball with terminology files in a web accessible directory. An example of such file is available here (this referes to reference.metoffice.gov.uk/data/wmo URIs and URLs, and links would appear as broken if installed elsewhere).

How to install the Terminology Manager

The current version of the terminology manager can be downloaded here a boundle of the rquired libs is available here (all the included jars must be made visibile prior or at command execution). This piece of software is self contained, requiring only a Java Run Time environment for execution. The software will need to operate on a triplestore whose file location is specified via command line (cfr. HOW-TO make a mapping and HOW-TO publish a terminology model). The Terminology Manager will need access to the file system both to access (r/w) triplestore files and to write/delete files in a directory served by the web server.
If the Terminology Manager is used to generate a mapping, a triplestore, if not present, will be initialized in the specified location.

How to install the mapping engine

 The mapping engine is in part contained in the Terminology Manager, and in part relies on the presence of D2RQ on the system. D2RQ, at its time, requires access to a relational engine with proper read-only permissions on a datatabase containg a representation of tables to be mapped. For installation of D2RQ please refer to the D2RQ documentation. Security settings are specific to the chosen relational engine.
